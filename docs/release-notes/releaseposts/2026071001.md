# CHILI GraFx Environment API: ETag caching, large output requests

![CHILI GraFx icon](/assets/icon-CHILI-GraFx.svg){.rn_icon}

## ✨ New & Improved

### ETag-based caching for the Environment API

An ETag is a version fingerprint the API returns with a resource. By sending it back on your next request via the `If-None-Match` header, the API can answer `304 Not Modified` instead of resending the full response — saving bandwidth and speeding up your integration. This release makes ETags more reliable and documents them properly:

- **ETags now update when related resources change** — for example, a change to a font style also refreshes the ETag of templates using it, so your cache never serves stale data.
- **Weak ETags are now handled correctly**, so caching keeps working when proxies or CDNs sit between your integration and the API.
- **ETags are now documented** in the Swagger reference and in a new [Caching with ETags](/GraFx-Developers/environment-api/04-making-api-calls/#caching-with-etags) section in the Developer Center.

### Improvements

- **Connector details are now cached privately** — the get connector endpoint returns `Cache-Control: private, max-age=60`, so repeated loads are faster while connector details stay out of shared caches and CDNs.

## 🐛 Bug Fixes

- Fixed `413 Content Too Large` errors on output requests for large documents: the 10MB size limit now applies only to the `variables` property instead of the entire request body, so projects built on big templates download reliably again.
